دليل التعاون: مجلدات، أدوات، Git، وماذا تشارك مع الفريق

خطة عملية لـ صبري · إسراء · حنين — بدون تكرار مجلدات المشروع، مع ربط واضح بـ توزيع الملفات، وتعليم رفع الكود على GitHub للمبتدئين.

١) هل أعمل من «مجلد جديد» لمشروع منفصل؟

الأفضل: لا. لا تنسخ المشروع كاملًا في مجلد جديد لكل شخص — ذلك يُضاعف النسخ ويصعّب الدمج ويخرج عن تاريخ Git.

الصحيح في الفريق الواحد:

  • مستودع واحد على GitHub: github.com/SabryAlaa10/mawid-plus
  • كل عضو يستنسخ (clone) المرة الأولى إلى مجلد على جهازه (مثلاً Desktop/mawid-plus).
  • للعمل على ميزة دون الخلط على main مباشرة: أنشئ فرعًا (branch) باسم واضح، ثم ارفعه وادمج عند الانتهاء (أو اسحب طلب دمج PR).

متى يُستخدم مجلد ثانٍ؟ فقط إذا احتجت نسخة تجريبية معزولة لاختبار كاسر — وليس كنسخة رئيسية للتطوير اليومي.

٢) ماذا «ترسل» لإسراء وحنين؟ (الأساسيات)

لا تعتمد على إرسال ملفات يدويًا عبر واتساب — المصدر الواحد هو GitHub.

٣) Cursor مقابل Android Studio — أي رأي؟

الويب (React / Vite)

Cursor (أو VS Code) مناسب جدًا: تشغيل npm run dev، اختبار، ودمج مع الذكاء الاصطناعي. لا حاجة لـ Android Studio.

أندرويد (Kotlin / Compose)

  • Android Studio ضروري لتشغيل المحاكي، Gradle بشكل موثوق، Layout Inspector، وتوقيع الـ APK. استخدمه كمرجع أساسي للبناء والتشغيل على جهاز أو محاكي.
  • Cursor يمكن استخدامه لقراءة/تعديل ملفات Kotlin إذا كان المشروع يبنى من الطرفية — لكن بدون Android Studio ستواجهون صعوبة في التشغيل والتصحيح لمعظم المطورين.

خلاصة عملية: مثالي تطوير أندرويد داخل Android Studio، ويمكن فتح نفس المجلد في Cursor لمساعدة الذكاء الاصطناعي على الملفات إن رغبتم — مع عدم حفظ تعارضات بين الأداتين (أغلقوا الملفات أو التزموا بأداة واحدة لنفس الجلسة عند التعديل المكثف).

٤) Git وGitHub للمبتدئين — من الصفر إلى الرفع

تثبيت الأدوات (مرة واحدة على Windows)

  1. ثبّت Git for Windows من الموقع الرسمي.
  2. سجّل دخولك إلى GitHub في المتصفح (نفس حساب المستودع أو دعوة Collaborator).

أول مرة: استنساخ المشروع

cd Desktop
git clone https://github.com/SabryAlaa10/mawid-plus.git
cd mawid-plus

قبل العمل: تحديث النسخة من الإنترنت

git checkout main
git pull origin main

فرع خاص بك (مُستحسن)

git checkout -b feature/hanin-android-theme

استبدل الجزء بعد feature/ بما يصف مهمتك (إنجليزي أو مختصر واضح).

بعد تعديل الملفات: حفظ خطوة في Git

git status
git add .
git commit -m "feat(android): تحسين ألوان الثيم في Color.kt"

قاعدة بسيطة لرسالة الـ commit:

  • feat — ميزة جديدة
  • fix — إصلاح خطأ
  • docs — توثيق فقط
  • chore — صيانة (مثلاً CI)
  • ثم نقطتان : ثم نطاق قصير ثم وصف بالعربي أو الإنجليزي.

رفع الفرع إلى GitHub

git push -u origin feature/hanin-android-theme

بعدها يظهر الفرع على GitHub — يمكن فتح Pull Request لمراجعة صبري قبل الدمج في main، أو الاتفاق على دمج مباشر للفريق الصغير.

إذا رفض الرفع (مصادقة)

استخدم Personal Access Token من GitHub → Settings → Developer settings → Tokens، كـ «كلمة مرور» عند الطلب، أو فعّل Git Credential Manager مع Windows.

٥) ماذا لا ترفع أبدًا؟

  • Mawid-Android/local.properties
  • Mawid-Web/.env
  • أي ملف يحتوي مفاتيح Supabase أو خرائط Google كاملة

هذه مذكورة في .gitignore في المستودع — التزموا بها.
