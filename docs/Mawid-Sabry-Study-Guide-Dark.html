٢) خريطة Supabase — كيف «يرتبط» المساعد بباقي النظام

عميل التطبيق (Android)

  • SupabaseProvider.kt ينشئ عميلاً واحداً بـ SUPABASE_URL + SUPABASE_ANON_KEY من BuildConfig، ويثبّت Auth وPostgrest.
  • المصادقة: Anonymous ثم Google (IDToken) — الملف: AuthRepository.kt.
  • جداول مباشرة: profiles (إدراج/تحديث صف المريض).
  • دوال SQL (RPC) عبر REST: /rest/v1/rpc/<name> لأن واجهة rpc في المكتبة قد لا تُعرض — نفس ما في تعليق الكود.

دوال RPC المهمة للمصادقة (مذاكرة الأسماء)

  • check_phone_registered(phone_e164) — هل الرقم مسجّل؟ (يُستدعى مع bearer anon في بعض المسارات).
  • claim_patient_profile_by_phone(phone_e164) — ربط المستخدم الحالي بصف الملف عند تسجيل الدخول.

الخادم (mawid-assistant)

  • يستخدم Service Role في database.py لقراءة/كتابة جداول مثل doctors وappointments (صلاحيات أعلى من التطبيق).
  • يتحقق من المستخدم عبر نفس JWT الذي أصدره Supabase Auth: إما طلب GET /auth/v1/user أو التحقق المحلي بـ SUPABASE_JWT_SECRET — الملف: auth.py.
تمييز مهم: التطبيق يتصل بـ Supabase مباشرة للهوية والملف الشخصي؛ أما شات المساعد فيتصل بـ ASSISTANT_API_BASE_URL (مثلاً FastAPI محلي) ويمرّر Authorization: Bearer <access_token> من جلسة Supabase — والسيرفر يثقّ بالتوكن عبر Supabase.

٥) مشروع المساعد — FastAPI mawid-assistant صبري

الترتيب: main.py (المسارات) → auth.py (الثقة بالتوكن) → routers/chat.py → services (llm, rag, specialty) → database.py (جداول Supabase بـ service role).

app/main.py

تسجيل الراوترات تحت /api + /health — اذكر لنفسك البادئات: الشات = /api/chat.

app/auth.py Supabase Auth API

  • الأولوية: GET {SUPABASE_URL}/auth/v1/user مع Authorization + apikey.
  • الاحتياطي: فك التوكن بـ SUPABASE_JWT_SECRET (HS256).

هذه هي الوصلة النظرية بين تطبيق Android والمساعد: نفس الـ JWT.

app/routers/chat.py

Depends(get_current_user) — بدون جلسة صالحة لا يدخل المستخدم للشات. يجمع RAG + LLM + اكتشاف تخصص + استدعاء get_doctors_by_specialty.

app/database.py Service Role + جداول

  • get_doctors_by_specialty — جدول doctors.
  • get_available_slots / book_appointment — doctors + appointments.

هنا لا يمرّ التطبيق مباشرة: السيرفر يملك مفتاحاً قوياً — فهم RLS على التطبيق مقابل service role هنا مهم جداً للمقابلة.

app/config.py + .env (غير مرفوع للـ git عادة)

المتغيرات: SUPABASE_URL، SUPABASE_SERVICE_ROLE_KEY / SUPABASE_ANON_KEY، SUPABASE_JWT_SECRET، ومفاتيح نماذج اللغة إن وُجدت.

ملفات خدمات للمذاكرة العميقة:

  • app/services/llm.py — قواعد الرد (بدون نصيحة علاجية خاطئة، إلخ).
  • app/services/rag.py — سياق المعرفة.
  • app/services/specialty.py — تطبيع التخصصات.
  • app/models/schemas.py — ChatRequest / ChatResponse.
