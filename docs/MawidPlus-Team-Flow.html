Mawid+ · Clinic Flow v3

توزيع الفريق ومسار العمل الخمسة

الثلاثة يشتغلوا على Mobile + Web. إسراء = Web (خدمات + صفحات) + Android (المستودعات) — منطق البيانات على المنصتين. حنين = Android (كل الشاشات + UI) + Web (هيكل). صبري = Supabase + AI + Web Auth + Android (DTOs/Models + Core).

١ المسؤوليات الإجمالية (مرجع سريع)

Sabry

صبري

  • Supabase كامل: تطبيق SQL، RLS، Realtime، Storage، اختبار السياسات.
  • Web: تسجيل دخول الطبيب، حماية المسارات، ملف الطبيب، الجدول، PDF، Toast، إشعارات.
  • Android: DTOs، Models، Splash، Navigation، الإشعارات، Home، AI/RAG كامل.
التوزيع: Web 35% · Android 35% · Supabase+AI 30%
Esraa

إسراء

  • Web: الخدمات (appointment, queue, doctor)، الـ hooks، المكونات (جداول، طابور، إحصاءات، رسوم)، والصفحات (Dashboard، Appointments، Queue).
  • Android: المستودعات (DoctorRepo, AppointmentRepo, QueueRepo, Result) + SupabaseProvider — نفس دور منطق البيانات على المنصتين.
التوزيع: Web 70% · Android 30%
حنين

حنين

  • Android بالكامل: تسجيل/دخول المريض، المستودعات (Doctor, Appointment, Queue)، البحث، التفاصيل، الخريطة، الحجز، مواعيدي، الطابور، الملف الشخصي، الثيم، المكونات المشتركة.
  • Web (هيكل فقط): DashboardShell، Sidebar.
التوزيع: Web 10% · Android 90%

التغييرات عن النسخة السابقة (v3):
١) الثلاثة يشتغلوا على Mobile: إسراء تاخد المستودعات (Repos) على Android — نفس دورها في Web (منطق البيانات).
٢) حنين تمسك كل شاشات Android (UI) — تستخدم المستودعات من إسراء وDTOs من صبري.
٣) صبري يمسك DTOs/Models + Core (Navigation, Splash, Notifications) + AI.
٤) إسراء: Web 70% + Android 30% | حنين: Android 90% + Web 10% | صبري: متوزع.
٥) كل مهندس له دور واضح في Android: إسراء = Data Layer، حنين = UI Layer، صبري = Models + Core + AI.

٥ طريقة رفع الشغل على GitHub (Git Workflow)

القاعدة الأساسية: لما تخلص أي جزء (ملف، مجموعة ملفات، أو مرحلة كاملة) → أنشئ branch → ارفعه → صبري يراجع ويعمل Pull Request → بعد الموافقة يُدمج في main.

لا أحد يرفع على main مباشرة — أبداً.
الخطوة ١ — قبل ما تبدأ شغل: اسحب آخر نسخة

في كل مرة قبل ما تبدأ شغل جديد، لازم تتأكد إن عندك آخر تحديثات من main:

  • # اذهب للفرع الرئيسي
  • git checkout main
  • # اسحب آخر التحديثات
  • git pull origin main
الخطوة ٢ — أنشئ فرع (Branch) جديد لشغلك

أنشئ branch باسم يوضح: اسمك + المرحلة + وصف قصير. أمثلة:

  • # الصيغة: اسمك/المرحلة/وصف-قصير
  • # مثال إسراء — خدمات الويب:
  • git checkout -b esraa/phase3/web-services-hooks
  • # مثال حنين — شاشات الحجز:
  • git checkout -b hanien/phase3/booking-screens
  • # مثال حنين — شاشات البحث:
  • git checkout -b hanien/phase3/search-screens
  • # مثال إسراء — المستودعات:
  • git checkout -b esraa/phase3/android-repositories
متى أعمل branch جديد؟
- لما تبدأ مجموعة ملفات مرتبطة ببعض (مثلاً: كل خدمات الويب مع بعض)
- مش لازم branch لكل ملف — اجمع الملفات المتعلقة ببعض في branch واحد
- لو الشغل كبير (أكتر من 10 ملفات)، قسّمه على أكتر من branch
الخطوة ٣ — اشتغل عادي وخلّص شغلك

اشتغل على الملفات المطلوبة منك. كل ما تخلص جزء، اعمل commit بوصف واضح بالعربي أو الإنجليزي:

  • # شوف الملفات اللي اتغيرت
  • git status
  • # أضف الملفات اللي خلصتها
  • git add src/services/appointmentService.js
  • git add src/hooks/useAppointments.js
  • # أو أضف كل الملفات المعدلة مرة واحدة
  • git add .
  • # اعمل commit بوصف واضح
  • git commit -m "feat: add appointment service and useAppointments hook"
  • # أمثلة رسائل commit جيدة:
  • # git commit -m "feat: add DoctorRepository with getDoctorById and search"
  • # git commit -m "feat: add BookingScreen and BookingViewModel"
  • # git commit -m "fix: handle CancellationException in AuthRepository"
  • # git commit -m "style: update theme colors and fonts"
قاعدة الـ Commit Message:
ابدأ بنوع التغيير: feat: (ميزة جديدة) · fix: (إصلاح خطأ) · style: (تصميم/ثيم) · refactor: (إعادة هيكلة)
ثم وصف قصير بالإنجليزي لما عملته.
الخطوة ٤ — ارفع الفرع على GitHub

بعد ما تخلّص كل الشغل المطلوب في الـ branch وعملت commits:

  • # ارفع الفرع على GitHub (أول مرة)
  • git push -u origin esraa/phase3/web-services-hooks
  • # لو عدّلت حاجة بعد كده وعايز ترفع تاني
  • git push
الخطوة ٥ — بلّغ صبري (على الجروب أو واتساب)

بعد الرفع، ابعت رسالة لصبري على الجروب بهذه المعلومات:

رسالة جاهزة (انسخها وعدّلها):

صبري، رفعت شغلي على GitHub:
- الفرع: esraa/phase3/web-services-hooks
- الملفات: appointmentService, queueService, doctorService, useAppointments, useQueue, useDoctor
- الوصف: خدمات الويب وhooks المواعيد والطابور والأطباء
- جاهز للمراجعة
الخطوة ٦ — صبري يراجع ويعمل Pull Request (مسؤولية صبري فقط)

صبري فقط — مراجعة الكود وإنشاء PR

العمل: صبري يراجع كود الفرع، يتأكد من الجودة، ويعمل Pull Request لدمجه في main.

  • # 1. اسحب الفرع محلياً وراجع الكود
  • git fetch origin
  • git checkout esraa/phase3/web-services-hooks
  • # 2. راجع التغييرات
  • git diff main...HEAD
  • # 3. لو الكود مظبوط — اعمل Pull Request من GitHub أو بالأمر:
  • gh pr create --base main --head esraa/phase3/web-services-hooks \
  • --title "Phase 3: Web services and hooks (Esraa)" \
  • --body "خدمات الويب (appointment, queue, doctor) مع hooks المواعيد والطابور"
  • # 4. بعد المراجعة النهائية — ادمج في main
  • gh pr merge --merge
  • # أو من موقع GitHub:
  • # اذهب إلى https://github.com/SabryAlaa10/mawid-plus/pulls
  • # اضغط "Merge pull request" ثم "Confirm merge"
لو صبري لقى مشكلة في الكود:
١) يكتب تعليق على الـ PR في GitHub يوضح المشكلة
٢) المهندس يعدّل الكود في نفس الـ branch ويرفع تاني (git push)
٣) صبري يراجع مرة تانية ← لو مظبوط يعمل merge
الخطوة ٧ — بعد الدمج: الكل يسحب التحديثات

بعد ما صبري يدمج PR في main، كل واحد في الفريق لازم يسحب التحديثات قبل ما يكمّل شغل:

  • # ارجع لـ main واسحب التحديثات
  • git checkout main
  • git pull origin main
  • # لو عندك branch شغال وعايز تحدّثه بتغييرات main الجديدة:
  • git checkout اسم-الفرع-بتاعك
  • git merge main
  • # لو ظهرت conflicts (تعارضات):
  • # 1. افتح الملفات المتعارضة في VS Code / Cursor
  • # 2. اختر التغيير الصحيح
  • # 3. بعد الحل:
  • git add .
  • git commit -m "merge: resolve conflicts with main"
