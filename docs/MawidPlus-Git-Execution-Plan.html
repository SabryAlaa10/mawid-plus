خطة تنفيذ كاملة لكل مهندس + GitHub (Branches & Pull Requests)

هذه الصفحة تلخّص “مين يعمل إيه” + “إزاي نشتغل صح على GitHub” طبقًا للتوزيع الحالي في MawidPlus-Team-Flow.html ومع قواعد عملية للمبتدئين في Git.

مسؤوليات كل مهندس الـ Workflow القياسي ما هو الـ Branch؟ إزاي تعمل Pull Request إعداد التشغيل خطة إعادة كتابة الكود

١) مسؤوليات كل مهندس (طبقًا للتوزيع الحالي)

صبري Lead Supabase تشغيل Web Auth + UI AI

  • Supabase (تشغيل على السحابة): تطبيق سكربتات migrations في SQL Editor، التحقق من RLS/Realtime/publication، ربط Storage لصور الأطباء، اختبار سياسات الطبيب/المريض.
  • Web: Login + ProtectedRoute + Auth + إشعارات الويب (TopBar/Notifications) + صفحات/واجهات (Profile/Schedule/Toast/Skeletons/Decor).
  • Android (Shell + Navigation): Routes/AppNavGraph + Splash + Notifications + WorkManager reminders + MainActivity/PatientApp + Home ربط زر المساعد لاحقًا.
  • AI: بناء RAG (مجلدات/ملفات جديدة لاحقًا) + تنسيق migrations 016/017 مع إسراء ثم تطبيقها على Supabase.

ملفات مرجعية: راجع القسم 2/3/4/5 في Team Flow.

إسراء Data Owner

  • منطق البيانات (Web): services + hooks + data components (appointments/queue/charts) + تنسيقات.
  • Android Data: SupabaseProvider + repositories + DTOs + models + Result/SeedDoctorIds.
  • Supabase (داخل Git): مراجعة/تحديث migrations كـ domain owner (لكن التشغيل على السحابة لصبري).

قاعدة: أي تغيير في schema/DTO/contracts لازم يتنسّق قبل الدمج.

حنين UI Owner

  • Android UI: Auth screens (login/register) + profile + theme + components + strings/resources.
  • Android Feature Screens: search/detail/map + booking + appointments + queue (تستهلك repositories إسراء).
  • Web الأساسية: DashboardShell/Sidebar + صفحات Dashboard/Appointments/Queue (تستهلك hooks/components إسراء).

قاعدة: حنين تجمع الواجهات، إسراء تضمن صحة البيانات.

٢) الـ Workflow القياسي اللي نمشي به (مختصر واضح)

  1. اسحب آخر نسخة من GitHub قبل أي شغل: git checkout main ثم git pull.
  2. اعمل Branch لميزة/إصلاح/إعادة كتابة: git checkout -b feature/… أو rewrite/….
  3. عدّل في نطاق ملفاتك فقط (حسب Team Flow).
  4. Commit صغير كل خطوة منطقية (لا تنتظر يوم كامل).
  5. Push للفرع: git push -u origin <branch>.
  6. افتح Pull Request → مراجعة → Merge إلى main.
  7. بعد الدمج ارجع main واعمل pull وكمّل.

٣) Branches يعني إيه؟ (باللغة العملية)

Branch = “مسار موازي” للكود بعيد عن main. تشتغل براحتك وتكسر وتجرب، وبعدين تدمج لما تستقر.

إحنا هنستخدم 3 أنواع أساسية:

  • feature/<name>-<scope> لميزة جديدة (مثال: feature/hanin-android-profile).
  • fix/<scope>-<issue> لإصلاح (مثال: fix/web-ci-env).
  • rewrite/<name>-<scope> لإعادة كتابة جزء بهدف الفهم (مثال: rewrite/esraa-repositories).

ممنوع: الشغل المباشر على main إلا في حالات طارئة وبموافقة صبري.

٤) Pull Request (PR) — يعمل إيه وإزاي نستخدمه؟

وظيفة الـ PR

  • مراجعة قبل الدمج (مافيش مفاجآت على main).
  • CI يشتغل تلقائيًا (ويب + أندرويد).
  • توثيق التغيير (عنوان + وصف + ملفات).

قواعد PR عندنا

  • صغير أفضل: PR كبير جدًا = صعب يتراجع.
  • ملفات مشتركة (AppNavGraph/Routes): لازم mention لصبري.
  • ما ترفعش أسرار: لا .env ولا local.properties.

خطوات فتح PR (ببساطة)

  1. بعد push افتح المستودع على GitHub.
  2. روح Pull requests → New pull request.
  3. Base = main وCompare = فرعك.
  4. اكتب عنوانًا واضحًا + وصف “ليه” + “إزاي أختبر”.
  5. Create PR ثم اطلب مراجعة من صبري (أو الشخص المناسب).

قالب وصف PR مقترح:

## Summary
- ...

## Test plan
- [ ] android: ...
- [ ] web: ...

٥) إعداد التشغيل لكل واحد (عشان يقدر يبني ويختبر)

Web (Mawid-Web)

cd Mawid-Web
npm install
copy .env.example .env   # على Windows
npm test
npm run dev

ملحوظة: CI بيستخدم قيم placeholder، لكن عند التشغيل المحلي محتاج .env فعلي.

Android (Mawid-Android)

cd Mawid-Android
# أنشئ local.properties في الجذر (غير مرفوع لـ Git)
./gradlew testDebugUnitTest
./gradlew assembleDebug

الأفضل تشغيل Android Studio للمحاكي والتصحيح.

٦) هدفك الحالي: “إعادة كتابة الكود للفهم” — نعملها إزاي من غير ما نبوّظ main؟

القاعدة الذهبية: main يفضل reference شغال. إعادة الكتابة تكون على فروع rewrite/… وتندمج على دفعات.

ليه؟ عشان نحتفظ بنسخة شغالة دايمًا + نسمح بالتعلّم بدون ضغط.

الطريقة المضمونة (لكل مهندس)

  1. ابدأ من main: git checkout main ثم git pull.
  2. اعمل فرع rewrite:
    • صبري: rewrite/sabry-web-auth-ui أو rewrite/sabry-ai-rag
    • إسراء: rewrite/esraa-data-layer
    • حنين: rewrite/hanin-android-ui أو rewrite/hanin-web-shell
  3. اقسم إعادة الكتابة إلى “وحدات صغيرة” (مثال: صفحة واحدة + ViewModel واحد + repo call واحد).
  4. كل وحدة: اختبر → commit → push → PR صغير.

نصيحة: إعادة الكتابة لا تعني copy/paste. اقرأ ثم اكتب بنفسك.

٧) قوالب أوامر جاهزة لكل واحد (انسخ/الصق)

إنشاء فرع + رفع أول مرة

git checkout main
git pull origin main
git checkout -b feature/<name>-<scope>
git add .
git commit -m \"feat(scope): short description\"
git push -u origin feature/<name>-<scope>

تحديث فرعك قبل ما تكمل

git checkout main
git pull origin main
git checkout <your-branch>
git merge main

لو حصل تعارض (conflict) اطلب مساعدة صبري فورًا.

٨) “مين يعمل إيه” كقائمة تنفيذ (نقطة بداية)

المهندس ابدأ بـ ناتج ملموس (Definition of Done)
صبري Supabase تشغيل + Web Login/ProtectedRoute دكتور يقدر يسجّل دخول + صفحات محمية + CI web أخضر
إسراء Data layer: repositories/DTO + web services/hooks استعلامات المواعيد/الطابور ثابتة + عقود بيانات واضحة + tests (إن وجدت) محدثة
حنين Android UI: login/register + web shell pages تدفق تسجيل مريض يعمل + صفحات لوحة تظهر بيانات من hooks بدون كسر

ممنوعات (تمنع مشاكل كبيرة)

  • لا ترفع Mawid-Web/.env أو Mawid-Android/local.properties.
  • لا تغيّر schema/DTO بدون تنسيق (إسراء + صبري).
  • لا تعمل PR ضخم جدًا مرة واحدة. قسمه.
